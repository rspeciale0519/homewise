### [--:--] Bulk drag-to-categorize + drag-overlay debugging saga (branch: feature/bulk-drag-categorize · commits: 44d0d81, c1246e1, ef1d452, 5d1054c · PR #39/#43/#45/#47)
**Synopsis:** Added cross-section bulk drag-to-categorize on the Document Library boards; most of the session was isolating why the dnd-kit drag preview rendered far from the cursor.
**What worked:** Fixed the offset by putting `setNodeRef` and `listeners` on the SAME element (the ~36px grip handle) — dnd-kit measures cursor offset within the `setNodeRef` element. For whole-card drag, moved attributes to the row root, added `stopPropagation` on inner action buttons' `onPointerDown`, and a `PointerSensor` ~5px activation distance so taps don't start drags. Centered the preview with `absolute left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2`. Kept the selection strip a fixed height so it stops reflowing. Evidence: shipped PR #39/#43/#45/#47; verified with programmatic mid-drag screenshots.
**What did NOT work:** `snapCenterToCursor` (`@dnd-kit/modifiers`) was a no-op — the handle container was already centered; the real bug was the 224px child preview overflowing inside the small container. The "overlay ~560px from cursor" symptom traced to `setNodeRef` on the 800px row but `listeners` on the right-edge grip. Only a programmatic mid-drag screenshot + pixel measurement (container center = exactly on cursor) isolated it.
