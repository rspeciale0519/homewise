### [15:50] Two more production-grade sync fixes: slow sequential writes + Inngest step-output limit (branch: feature/mls-go-live · commits: none)
**Synopsis:** Backfill attempt 2 surfaced two more real defects. (1) Sequential per-listing upserts ran ~0.8/s (13k listings ≈ 5h; full Stellar feed would be days). (2) Run died at ~812 listings: Inngest "step output size is greater than the limit" — `fetch-property-page-N` returned the raw 200-listing RESO page (with Media arrays) as memoized step output; a heavy page blew the cap. Same failure would occur in Inngest Cloud (4MB).
**What worked:** (1) `processPropertyPage` now writes in `Promise.all` chunks of 10 (`PAGE_WRITE_CONCURRENCY`) → throughput jumped to ~7/s mid-run via hot reload. (2) Merged fetch+process into one step `sync-property-page-N` returning only small aggregates `{processed, upserted, deleted, maxCursor, nextLink}`; retry refetches the page; upserts idempotent. Applied same pattern to `sync-openhouse-page-N`. Stuck `status="syncing"` didn't block resume because only cron invocations skip on syncing — event-triggered run resumed from persisted cursor (~800 listings in). Evidence: vitest 5/5 + `tsc --noEmit` clean; listings climbed past 812 post-resume.

Re-armed monitor now has explicit stall detection (6×60s no change → alert).
